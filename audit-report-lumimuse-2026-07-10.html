📝 总体评估

📋 总体评估

项目概况：LumiMuse 是 Next.js 16 + React 19 + strict TypeScript + SQLite 的本地优先 AI 角色聊天应用。当前提交的 lint、443 项测试、回归脚本和生产构建全部通过，数据层事务、记忆队列恢复、SSRF 双层检查与前端竞态防护体现出较成熟的工程基础。

主要风险：首要问题是手工跨源重定向会转发 Authorization，存在真实 API 密钥泄露路径。其次是 SSRF 特殊地址覆盖、代理信任限流、后台模型/生图超时、长会话全量加载，以及前端键盘可达性和测试真实性缺口。

亮点：本次没有确认数据损坏、SQL 注入、路径穿越、鉴权绕过或重复队列写入；SQLite 事务和迁移测试、密钥掩码、上传边界、私有文件缓存、Node 20/24 CI 矩阵均表现良好。

优先修复：先阻断跨源凭据转发；随后统一所有外部请求 deadline、补齐 SSRF 地址判定与可信代理解析；再补 DOM/浏览器行为测试并修复三个键盘工作流；最后增加 Docker smoke/readiness。

总评：6.9 / 10，B。代码并非“不可维护山体”，但安全边界中存在一个 High，测试与发布仍有成体系的 Medium 风险，不宜把当前全绿 CI 等同于完全发布就绪。

🔍 详细发现

1. 跨源重定向会把上游 API 密钥转发到新目标

HighRemediated
📁 Security📍 src/lib/ssrf-guard.ts:303-335; src/lib/api-client.ts:127-135,223-231🎯 safeFetch / chatCompletion
问题：safeFetch 手工处理 3xx 时复用原始 init；当 Location 改变 origin，Authorization、Cookie 等敏感头仍被原样发送。
影响：可信供应商的开放重定向、被攻陷服务或错误配置即可把 LLM/Embedding/Reranker 凭据泄露给第三方。
现实场景：带 Bearer 密钥请求可信上游 → 上游返回跨源 302 → 新公网目标通过 SSRF 校验 → 应用携带原 Authorization 请求攻击者域名。
文件：src/lib/ssrf-guard.ts:303-335; src/lib/api-client.ts:127-135,223-231 函数 / 模块：safeFetch / chatCompletion 实际行为：redirect:'manual' 后重新调用 undiciFetch(currentUrl,{...init})，跨 origin 前没有清除 credential headers。
最小修复
默认拒绝跨源重定向；或在 origin 变化时删除 authorization、proxy-authorization、cookie 等凭据头。
⏱ 工作量： 2–4 小时  ·  回归测试： 两个测试服务模拟 A→B 302，断言跨源跳转不携带 Authorization；覆盖相对 Location、协议变化与多跳。

2. SSRF 分类遗漏 100.64.0.0/10 等非公网地址段

MediumRemediated
📁 Security📍 src/lib/ssrf-guard.ts:27-42,157-170🎯 classifyIp / assertSafeUrl / createGuardedAgent
问题：DNS 与 socket 两层都复用零散黑名单，未拒绝 CGNAT/Tailscale 常见的 100.64.0.0/10，并可能遗漏其它特殊用途地址。
影响：部署位于 overlay/VPN 或特殊路由网络时，用户可配置上游 URL 可能访问本不应暴露的内部服务。
现实场景：攻击者控制供应商 URL 指向 100.64/10 内部服务 → 两层分类均判定安全 → 服务端携带请求访问内部网络。
文件：src/lib/ssrf-guard.ts:27-42,157-170 函数 / 模块：classifyIp / assertSafeUrl / createGuardedAgent 实际行为：永久禁止与私网列表未覆盖 100.64.0.0/10；预解析和真实 socket 校验使用同一分类函数。
最小修复
改为仅允许 globally routable 地址的完整判定，并为 IPv4/IPv6 特殊地址段建立表驱动测试。
⏱ 工作量： 4–8 小时  ·  回归测试： 覆盖 CGNAT、multicast、benchmark、documentation、reserved、ULA/site-local 等地址与 ALLOW_LOCAL_NETWORK 组合。

3. TRUST_PROXY=1 时无条件信任 X-Forwarded-For 首项

MediumRemediated
📁 Security📍 src/app/api/auth/route.ts:16-18,28-42🎯 getClientIp / login rate limit
问题：开启 TRUST_PROXY 后直接把 X-Forwarded-For 第一项作为限流键，没有验证请求来源属于可信代理，也没有按可信跳数解析。
影响：应用仍可直连或代理保留客户端头时，攻击者可每次伪造不同 IP，绕过每桶 10 次的登录失败限制。
现实场景：部署开启 TRUST_PROXY 且端口可直达 → 攻击者轮换 XFF 首项 → 每次进入新桶 → 在线猜测访问密码。
文件：src/app/api/auth/route.ts:16-18,28-42 函数 / 模块：getClientIp / login rate limit 实际行为：getClientIp 在开关开启时直接 split(',')[0]，没有可信代理 CIDR/跳数判断。
最小修复
仅对来自配置可信代理 CIDR 的请求解析转发链，并从右向左剥离可信代理；否则使用共享桶。
⏱ 工作量： 3–6 小时  ·  回归测试： 覆盖直连伪造 XFF、可信单层代理、代理追加、多跳和不可信来源。

4. 后台 LLM 调用没有服务端 deadline

MediumRemediated
📁 Stability📍 src/lib/api-client.ts:200-230; src/lib/memory-engine.ts:574-579; src/lib/memory-profile.ts:381-413; src/app/api/image-gen/prompt/route.ts:215-218🎯 chatCompletion / extractMemories / generateMemoryProfilePatch
问题：chatCompletion 只有调用方 signal，没有默认 watchdog；记忆提取、画像更新和生图提示词调用显式不传 signal。
影响：上游建立连接后不返回时，任务 Promise 可无限等待，记忆队列或画像更新无法继续并持续占用网络资源。
现实场景：后台模型半开连接 → extractMemories 永不返回 → 当前任务长期 processing，后续记忆提取停滞直至进程恢复。
文件：src/lib/api-client.ts:200-230; src/lib/memory-engine.ts:574-579; src/lib/memory-profile.ts:381-413; src/app/api/image-gen/prompt/route.ts:215-218 函数 / 模块：chatCompletion / extractMemories / generateMemoryProfilePatch 实际行为：三条后台调用传 undefined signal；safeFetch 文档明确不提供全局超时。
最小修复
合并调用方 signal 与可配置后台 watchdog；超时后任务进入 failed/可重试并释放 processing 标志。
⏱ 工作量： 2–4 小时  ·  回归测试： mock 仅在 abort 后拒绝的 safeFetch，断言 deadline 到达后任务失败且队列能处理下一项。

5. NovelAI 与 ComfyUI 请求未真正接入取消和超时

MediumRemediated
📁 Stability📍 src/app/api/image-gen/route.ts:283-358,430-484,628-636🎯 generateNAI / generateComfyUI
问题：NAI 不接收 request.signal；ComfyUI 只在轮询前后检查 aborted，但 /prompt、/history 与最终下载 fetch 本身未传 signal，卡住时 120 秒上限无效。
影响：用户离开或上游半开连接后请求仍可长期存活，重复生成还会增加付费调用和资源占用。
现实场景：ComfyUI /history 永不响应 → 循环无法再次检查时间或 abort → 请求超过 maxWait；NAI 同样没有 deadline。
文件：src/app/api/image-gen/route.ts:283-358,430-484,628-636 函数 / 模块：generateNAI / generateComfyUI 实际行为：相关 safeFetch 均无 signal；对比 SD/custom 已使用 withTimeoutSignal(resolveGenerateTimeout)。
最小修复
所有生图引擎统一组合 request.signal 与 generate_timeout_ms，并把同一 deadline 传到提交、轮询和下载。
⏱ 工作量： 3–6 小时  ·  回归测试： 分别让 NAI、/prompt、/history、/view 永不响应，验证客户端 abort 与配置超时均能终止。

6. 聊天热路径为寻找 summary 仍加载整段会话

MediumRemediated
📁 Performance📍 src/lib/chat-engine.ts:515-533🎯 runChat
问题：每轮聊天先 SELECT * 读取会话全部 messages，再在内存中寻找最后 summary 并切片；metadata 与旧正文也全部进入内存。
影响：长会话每轮延迟、内存与序列化成本线性增长，summary 本应提供的历史边界没有下推到查询层。
现实场景：角色积累数千条消息 → 每次发送都加载全历史，即使实际上下文只需最后 summary 之后几十条。
文件：src/lib/chat-engine.ts:515-533 函数 / 模块：runChat 实际行为：SELECT * FROM messages WHERE conversation_id=? 无 seq 下界或 LIMIT，随后 findLastIndex/slice。
最小修复
先查询最后 summary 的 seq，再按 seq 下界获取上下文；提取和管理 API 统一使用分页仓储。
⏱ 工作量： 0.5–1 天  ·  回归测试： 构造 5000 条历史与中间 summary，断言上下文查询只读取 summary 后窗口且结果行为不变。

8. 三个关键 UI 工作流缺少完整键盘与焦点语义
